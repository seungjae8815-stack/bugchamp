# 서버 권위(Server-Authoritative) 전환 설계

> 결정일 2026-07-20 (2026-07-20 개정). **재화·진행도를 서버가 소유**한다.
> 목적: 클라이언트 조작으로 재화·트로피를 만들어내는 것을 원천 차단 +
> **PvP 심화의 토대**(랭킹·대전은 어차피 온라인이고, 출시 후 이관은 훨씬 비싸다).
>
> **개정 요지**: 로그인 전면 필수화를 **철회**한다. 익명 인증도 서버상 진짜 계정이라
> 서버 권위에 충분하다. 로그인은 **구매 시에만** 요구한다(사유는 §5-2).
> 따라서 콘솔 재제출·방침 수정이 불필요하고 온보딩 마찰도 없다.

---

## 1. 왜 하는가

현재는 진행도가 기기(Hive)에 있고 클라이언트가 자기 세이브를 직접 쓴다.
영수증 검증(§backend_supabase.md §9)을 붙여 **위조 영수증**은 막았지만,
앱을 개조해 결제 흐름을 건너뛰고 재화를 넣는 것은 여전히 가능하다.

재화를 서버가 들고 있으면 클라이언트가 무엇을 조작하든 의미가 없어진다.

## 2. 핵심 제약 — 게임 로직은 Dart 로 3,700줄이다

| 패키지 | 줄 수 | 내용 |
|---|---|---|
| `core_models` | 1,277 | 개체·종·사이즈 롤·오행 |
| `core_run` | 1,904 | 방치 수학·오프라인 보상·각종 config |
| `core_battle` | 383 | 전투 시뮬레이션(결정론) |
| `core_gathering` | 164 | 레거시 |

**이걸 TypeScript 로 다시 짜면 안 된다.** 밸런스 JSON 을 하나 고칠 때마다
Dart 와 TS 양쪽을 맞춰야 하고, 어긋나는 순간 "클라에선 되는데 서버가 거부"가
발생한다. 결정론 전투는 특히 치명적이다 — 한쪽이라도 다르면 결과가 갈린다.

### 결론: **서버도 Dart 로 돌린다**

`core_models` / `core_run` / `core_battle` 을 **그대로 import** 하는 Dart 서버를 둔다.
로직이 한 벌이므로 드리프트가 구조적으로 불가능하다.

- 런타임: Dart + `shelf`
- 호스팅: **Google Cloud Run** (이미 GCP 프로젝트 `bugchamp` 사용 중, Play 연동됨)
- DB: **기존 Supabase Postgres 유지** (auth·테이블 이미 있음)
- 인증: 클라이언트가 보낸 **Supabase JWT 를 서버가 검증**
  (**ES256 비대칭** — JWKS 공개키로 검증하므로 서버에 JWT 비밀이 필요 없다.
  이 프로젝트는 2026-07-15경 레거시 HS256 에서 ECC P-256 으로 전환됨)

> Supabase Edge Function 은 Deno(JS)라 Dart 코드를 못 쓴다.
> `dart compile js` 로 우겨넣는 방법도 있으나 디버깅·의존성이 지저분해진다.
> 영수증 검증 함수만 Edge Function 으로 남기고, 게임 로직은 Dart 서버로 간다.

## 3. 방치형 특유의 문제와 해법

핵심 루프가 "초당 골드가 쌓인다"이다. 탭마다 서버를 부를 수 없다.

### 해법: 서버가 **다시 계산**한다 (클라 보고를 믿지 않는다)

방치 수입은 `(스탯, 스테이지, 경과시간)` 의 **결정론적 함수**다.
따라서 서버는 클라이언트가 "500골드 벌었다"고 말하는 걸 믿을 필요 없이,
`lastSeen` 부터 지금까지를 **직접 계산**하면 된다. 이미 `computeOfflineReward`
가 그 일을 한다 — 같은 함수를 서버에서 쓴다.

수동 탭은 상한을 둔다: "경과 t초 동안 최대 `t × 최대탭속도` 회".

### 액션 목록 (서버 RPC 가 될 것들)

| 액션 | 서버가 검증하는 것 |
|---|---|
| `sync` | 경과시간으로 방치 수입 재계산, 탭 수 상한 검사 |
| `upgrade` | 비용 차감 가능 여부 |
| `battle` | `core_battle.simulate` 를 **서버에서** 돌려 결과·보상 확정 |
| `breed` / `incubate` / `train` | 재료·타이머 조건 |
| `purchase` | 영수증 검증(기존 §9) 후 지급 |
| `adReward` | 광고 보상 — 서버 서명 검증(SSV) 필요 |

## 5. 이 전환으로 **깨지는 것들** (반드시 알고 시작할 것)

### 5-1. 오프라인은 "보기만" 가능
연결이 필요한 것: 구매·전투·강화 등 **재화가 변하는 액션** + 주기적 sync.
연결 없이도 되는 것: 화면 보기, 채집 연출, 보유 곤충 구경.

방치의 재미는 *시간이 가치를 만든다*는 것이지 *앱이 꺼진 채 실행된다*가 아니다.
오프라인 보상은 경과시간의 함수이므로 재접속 시 서버가 계산해 주면 동일하다.

### 5-2. 익명 계정은 잃어버리면 끝 — **결제와 얽히면 심각**
익명 uid 는 기기에만 있다. 앱 삭제·기기 변경 시 서버에 세이브가 있어도
주인을 찾을 수 없다(지금도 동일).

문제는 결제다. `verified_purchases` 는 영수증 토큰을 uid 에 묶으므로,
익명 유저가 재설치하면 **"구매 복원"을 눌러도 그 영수증은 죽은 uid 소유**라
`owned_by_other` 로 거부된다 → **돈은 냈는데 복구 불가**.

**해법: 구매 시에만 로그인 요구.**

| | 익명 | 구글 로그인 |
|---|---|---|
| 채집·전투·랭킹·짝짓기 | ✅ | ✅ |
| 서버 권위(치팅 차단) | ✅ | ✅ |
| **상점 구매** | ❌ 로그인 유도 | ✅ |
| 기기 변경·재설치 복구 | ❌ | ✅ |

안 사는 유저는 마찰 0(지금과 동일). 사는 유저는 어차피 구글 계정으로
결제하므로 로그인이 자연스럽고, **산 것을 잃지 않는다.**

### 5-3. 서버 다운 = 게임 전체 정지
지금은 서버가 죽어도 게임이 돌아간다. 앞으로는 아니다.
Cloud Run 은 관리형이라 안정적이나, **장애 시 전면 중단**을 받아들여야 한다.

### 5-4. 체감 지연
모든 액션이 왕복한다. 완화: **낙관적 UI**(먼저 그리고 서버 확정 시 보정).
보정이 틀리면 화면이 되돌아가므로 어긋남 처리 설계가 필요하다.

### 5-5. 비용
Cloud Run + Supabase. 유저가 적을 땐 거의 무료지만, 방치형은
**접속 중 주기적 sync** 가 필요해 요청 수가 많다. sync 주기 설계가 곧 비용이다.

### 5-6. ~~콘솔·법무 재제출~~ → **불필요**
로그인 전면 필수화를 철회했으므로 그대로 유지된다:
- **앱 액세스 권한**: "모든 기능 제한 없이 사용 가능" 유지
  (구매는 기능 제한이 아니라 선택 구매다)
- **개인정보처리방침**: "구글 로그인은 선택" 문구 유지
- **데이터 보안**: 이메일은 계속 **선택** 수집

> 단, **채팅**을 넣었으므로 콘텐츠 등급("사용자 간 소통")과 데이터 보안
> (메시지 내용)은 별건으로 갱신해야 한다.

### 5-7. 기존 테스터 데이터
지금 로컬 세이브를 가진 기기가 있다. 최초 로그인 시 **1회 업로드**로 이관하고,
이후 로컬 쓰기를 막는다. 이관 시점 이후의 로컬 변경은 버린다.

### 오프라인 정책이 관대해도 되는 이유

서버가 **경과시간으로 다시 계산**하므로 클라이언트가 오프라인에서 골드가
오르는 걸 그려도 보안상 손해가 없다 — 서버는 클라 숫자를 애초에 안 믿는다.
게다가 서버도 같은 `core_run` 함수를 쓰므로 **클라 예측과 서버 확정값이
거의 일치**한다(화면 되돌아감이 드물다).

→ §5-1 의 "오프라인 플레이 불가" 경고는 이 설계에서 **과했다**.
   온라인이 필요한 것은 재화가 실제로 변하는 액션(강화·전투·구매·짝짓기)뿐이다.

## 6-2. 🔴 P2 만으로는 동작하지 않는다 (2026-07-20 실기에서 확인)

배포 후 실제로 전투를 걸어보고 확인한 사실:

```
12:25:50  POST /state   빈 세이브 이관 (곤충 0마리)
12:29     개발자 모드로 곤충 추가  ← 로컬에만 반영
12:31:51  POST /battle  400  (서버 세이브엔 그 곤충이 없음)
```

**원인**: 채집·강화·짝짓기·부화 등 나머지 액션이 아직 로컬 세이브를 고친다.
서버는 그 변경을 모르므로 세이브가 즉시 낡고, 전투·구매가 전부 거부된다.

**결론: P3 는 선택이 아니라 P2 를 쓸 수 있게 만드는 필수 조건이다.**
전환이 끝날 때까지 `GAME_SERVER_URL` 을 주입하지 않는 빌드를 배포해야 한다
(그러면 기존 로컬 경로로 정상 동작한다).

임시로 "로컬 세이브를 주기적으로 서버에 올리기"를 넣고 싶을 수 있는데,
**그건 하면 안 된다** — 클라이언트가 상태를 지시하게 되어 서버 권위가
통째로 무의미해진다. 지금 막고 있는 바로 그 공격 경로다.

## 6-4. 수동 전투는 왜 시드를 안 주는가 (2026-07-20)

자동 전투는 서버가 시드를 주고 앱이 같은 `core_battle` 로 **재현**해 연출한다.
수동 전투는 그럴 수 없다 — 시드를 알면 상대가 매 라운드 무엇을 낼지 미리
계산해 항상 이기는 수를 고를 수 있다. **심리전이라는 장르 자체가 사라진다.**

그래서 서버가 세션을 들고, 한 수마다 그 라운드 결과만 돌려준다.
앱은 `ManualBattleDriver` 만 알고 전투가 어디서 도는지 모른다:

| 드라이버 | 쓰는 곳 | 보상 |
|---|---|---|
| `LocalManualDriver` | 야생 상대 | 화면이 `pvpReward` 로 지급 |
| `ServerManualDriver` | 실제 유저 상대 | **서버가 이미 지급**(`rewardsApplied`) |

`rewardsApplied` 가 이 설계의 핵심이다. 이게 없으면 서버가 준 보상 위에
화면이 또 지급해 **두 배**가 된다.

서버가 한 수를 거부하거나 불통이면 앱은 **로컬로 대신 계산하지 않고** 전투를
중단한다. 폴백하면 서버 권위가 무의미해지고 화면과 서버 상태가 갈린다.

---

## 6-5. 야생 상대는 왜 서버가 만드는가 (2026-07-20)

야생(합성) 상대는 내 로스터 평균 × 티어 배율로 만들어진다. 앱이 만들면
**약한 상대를 만들어 트로피를 쓸어담을 수 있다.** 티어도 id 로만 받는다 —
배율을 직접 보내게 하면 같은 구멍이 그대로 남는다.

서버가 만들면 새 문제가 생긴다: 앱이 그리는 상대와 서버가 싸운 상대가
다를 수 있다. 그래서 서버가 **자기가 싸운 상대를 응답에 실어** 준다
(`foe`: 종 id·오행·기질·선호 스탠스·스탯). 앱은 이걸로 그리고, 자동
전투는 이걸로 재생한다.

전투 장소도 같은 규칙이어야 한다 — 양쪽 다 **상대 리드 곤충의 오행**.
앱이 `scout.location`(자기가 만든 상대 기준)을 쓰면 장소 상성이 달라져
같은 시드로도 다른 결과가 나온다.

## 6-6. 방치 진행을 서버가 확정한다 (2026-07-21)

`adoptServerSave` 는 15초마다 세이브를 통째 교체한다. 그래서 클라가 로컬로
올린 스테이지·미션·선물·보상은 다음 sync 에 **덮여 사라졌다**(재시작 시
스테이지 진행 손실, 방치수입을 낮은 스테이지로 계산). 서버로 옮겨 고쳤다.

- **스테이지**: `simulateIdleProgress`(core_run 공용)로 sync 가 경과시간에
  스테이지를 올린다. 적 HP 가 커지며 처치가 느려져 전투력에 수렴한다.
  → 스테이지가 시간/전투력 기반이 됐다(사장님 합의).
- **미션 진행**: sync(처치)·upgrade(강화)에서 활성 미션을 bump. 진행도를
  서버가 소유하므로 조작 수령 불가.
- **선물 스폰**: sync 가 시각 기반으로(서버 RNG). 클라 타이머는 없앴다.
- **수령**: `/mission/claim`·`/gift/claim`·`/daily/claim`·`/roadmap/claim`.
  일일보상은 UTC 날짜당 1회(시간 게이트는 UI — 서버는 타임존을 모른다).

앱: 서버 모드에서 아이들 루프는 **연출만** 돈다. applyReward(idle)·
reachStage·maybeSpawnGift·startup 오프라인 계산을 서버 모드에서 억제하고,
스크롤러 `_stage` 는 서버 스테이지 아래로 안 내려간다(하한).

**호출 증가 없음**: 무거운 계산은 전부 기존 15초 sync 에 얹었다. 선물
타이머가 없어져 오히려 순 감소. 새로 느는 건 사용자 수령 탭뿐.

⚠️ `/gift/claim` 의 광고 배수(`doubled`)와 광고 보상 젤리는 아직 클라
신뢰다 — AdMob SSV(출시 후)가 붙어야 서버가 "실제로 봤는가"를 확인한다.

## 7. 되돌릴 수 있는가

**지금은 완전히 되돌릴 수 있다.** 앱은 `GAME_SERVER_URL` 이 주입된 경우에만
서버 경로를 쓰고, 없으면 기존 로컬 계산 그대로다. 서버를 안 붙이고 빌드하면
지금까지의 변경이 사용자에게 보이지 않는다.

단, **서버 경로가 켜지면 실패 시 로컬로 폴백하지 않는다** — 폴백하면
"서버가 거부하면 로컬로 처리"가 되어 권위가 무의미해지기 때문이다.
연결이 안 되면 사용자에게 알리고 액션을 취소한다.
